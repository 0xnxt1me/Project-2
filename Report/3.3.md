## Correcao antes do fuzzing

O código original continha um bug crítico na função pwm_handle_command (commands.c) que fazia o programa colapsar (crash) logo no primeiro comando, deixando o fuzzer com 0.00% de cobertura.
O Problema: Sombreado de Variáveis (Variable Shadowing)
```bash
int i; // 1. O 'i' original (sem valor)

for (size_t i = 0; i < N_AVAILABLE_COMMANDS; i++) { // 2. NOVO 'i' local ao ciclo
  if (strcmp(command, AVAILABLE_COMMANDS[i].command_name) == 0) { break; }
} // 3. O 'i' local morre aqui!

r = AVAILABLE_COMMANDS[i].command_func(...); // 4. Usa o 'i' original COM LIXO DE MEMÓRIA!
```
Ao colocar size_t i dentro do for, o C criou uma variável temporária. O índice real do comando foi destruído ao sair do ciclo. Na linha de execução da função, o programa usou o i do topo, que por nunca ter sido inicializado continha um valor "lixo" da memória Stack, originando um Segmentation Fault imediato em qualquer execução.
A Solução Implementada

Eliminou-se a dupla declaração e adicionou-se uma validação de limites para rejeitar comandos inválidos com elegância, permitindo que o fuzzer explore o resto do código:

```bash
pwm_res_t pwm_handle_command(PWM* pwm, char* command, char** command_args, int n_args) {
  pwm_res_t r = PWM_OK;
  size_t i; // Declaração única no topo
  int j;

  for (i = 0; i < N_AVAILABLE_COMMANDS; i++) {
    if (strcmp(command, AVAILABLE_COMMANDS[i].command_name) == 0) { break; }
  }

  // SALVAGUARDA: Se o comando não existir (ex: "adlp"), aborta em segurança
  if (i == N_AVAILABLE_COMMANDS) {
    pwm_error("Unrecognized command: '%s'!", command);
    return PWM_OPERATION_NOT_RECOGNIZED;
  }

  // Execução 100% segura com o índice correto
  r = AVAILABLE_COMMANDS[i].command_func(pwm, n_args, command_args);
  ```

Correcão, devemos de mover a linha de diagnóstico do printf para dentro do bloco protegido que garante que a estrutura foi devidamente validada pelo assert_state_and_args:

```C
static pwm_res_t list_command(PWM* pwm, int n_args, char** args) {
  pwm_res_t r = assert_state_and_args(pwm, PWM_INIT_STATE, 0, n_args);
  if (r == PWM_OK) {
    printf("In memory-contents for '%s'\n", (*pwm)->file); // Mover para aqui
    int count = 0;
    r = pwm_iterate(*pwm, &list_iterator, &count);
    if (r == PWM_OK) {
      printf("%d users\n", count);
    } 
  }
  return r; 
}
```

Todos os ficheiros base executados, como blab1.txt, blab2.txt desencanderam este erro, quando o comando init é invocado, a função pwm_alloc reserva memória para o nome do ficheiro usando pwm->file = strdup(file);. 
O LeakSanitizer acusa que essa memória de 13 bytes fica "órfã" (perdida) no encerramento da aplicação. Isto acontece porque na função main (pwm.c) ou ao processar comandos, em determinados fluxos de saída (como o encerramento por detetar Fim de Ficheiro EOF), o recurso opaco do ponteiro PWM não é desalocado chamando explicitamente a rotina pwm_free.

```bash
==46724==ERROR: LeakSanitizer: detected memory leaks / Direct leak of 13 byte(s) in 1 object(s)
```

Para isso corrigimos a funcao pwm_free no ficheiro core.c

```bash
pwm_res_t pwm_free(PWM pwm) {
  if (pwm == NULL) return PWM_OK; // Salvaguarda anti-crash

  pwm_node_t* node = pwm -> entries;
  while (node != NULL) {
    pwm_node_t* aux = node;
    node = node -> next;
    free(aux);
  }
  
  // CORREÇÃO AQUI: Liberta a string do ficheiro que o strdup alocou!
  if (pwm -> file != NULL) {
    free(pwm -> file);
  }

  free(pwm);
  return PWM_OK;
}
```

Na funcao pwm_interactive_session no commands.c corrigimos isto também:

```bash
if (pwm != NULL) {
    pwm_free(pwm);
    pwm = NULL; // Evita ponteiros pendentes
  }
}
```
Pois nafunção (por volta da linha 279), o código original já tenta fazer pwm_free(pwm);.

No entanto, por causa da forma como os comandos save ou o encerramento por EOF funcionam, o ponteiro local pwm pode ficar desalinhado.
