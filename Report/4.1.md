# 4.1 Design Vulnerabilities

## 1. Admin Position & Tampering Flaw in `pwm_open`

### Vulnerability Explanation
If an adversary has write access to the credential database file (e.g., `secret.txt`), they can exploit the previous design of the password manager in two severe ways:

1. **Admin Password Verification Bypass / Account Hijacking:**
   - In `pwm_open()`, the program verified the master admin password by checking the credentials of the entry loaded on the very first line of the database (`count == 1`).
   - However, the function **never validated** that the username on this first line was actually `"admin"`.
   - An adversary could swap the first line with another user they controlled. When the administrator attempted to open the database file, `pwm` would verify the typed password against the adversary's salt and hash. Furthermore, the head of the loaded list (`entries`) would be owned by a different user.

2. **Admin Account Deletion:**
   - The password manager is designed to prevent deleting the special admin user (`PWM_ADMIN_USER`). In `pwm_delete()`, a check is made to disallow the deletion:
     ```c
     if (strcmp(user, PWM_ADMIN_USER) == 0) {
       pwm_error("User '%s' cannot be deleted!", user);
       r = PWM_OPERATION_NOT_ALLOWED;
     }
     ```
   - However, the actual traversal loop for node deletion traverses the linked list starting at `entries` and checks `node->next->user`:
     ```c
     while (node -> next != NULL) {
       if (strcmp(node -> next -> user, user) == 0) {
         break;
       }
       node = node -> next;
     }
     ```
   - Because this loop checks `node->next`, **the first node (`entries`) can never be matched or deleted**.
   - If `"admin"` is in its correct first position, it is protected from deletion because it resides at the head. But if an adversary alters the position of `"admin"` (e.g., placing a dummy user at the head and moving `"admin"` to the second line), `node->next` can now match `"admin"`, allowing `pwm_delete()` to successfully remove the admin user from the memory and database.

---

### Solution / Code Fix

To address these vulnerabilities, we modified the `pwm_open` function in `core.c`:

1. **Enforcing Admin as the First Entry:**
   When parsing the first line (`count == 1`), we explicitly assert that the user ID is exactly `"admin"`. If it is not, we immediately abort, free the allocated node (to prevent memory leaks), and return `PWM_FILE_CORRUPT`.
   ```c
   if (count == 1) {
     if (strcmp(node->user, PWM_ADMIN_USER) != 0) {
       r = PWM_FILE_CORRUPT;
       pwm_error("Corrupt file '%s': first user must be '%s'!", (*pwm)->file, PWM_ADMIN_USER);
       free(node);
       break;
     }
     ...
   }
   ```

2. **Rejecting Empty Files:**
   If the database file is completely empty, `count` remains `0`. We added a check at the end of the load loop to verify `count > 0`, otherwise returning `PWM_FILE_CORRUPT` and closing the file.
   ```c
   if (r == PWM_OK && count == 0) {
     r = PWM_FILE_CORRUPT;
     pwm_error("Corrupt file '%s': empty password database!", (*pwm)->file);
   }
   ```
