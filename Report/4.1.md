# 4.1 Design Vulnerabilities

## 1. Admin Position & Tampering Flaw in `pwm_open`

### Vulnerability Explanation
If an adversary has write access to the credential database file (e.g., `secret.txt`), they can exploit the previous design of the password manager in two severe ways:

1. **Admin Password Verification Bypass / Account Hijacking:**
   - In `pwm_open()`, the program verified the master admin password by checking the credentials of the entry loaded on the very first line of the database (`count == 1`).
   - However, the function **never validated** that the username on this first line was actually `"admin"`.
   - An adversary could swap the first line with another user they controlled. When the administrator attempted to open the database file, `pwm` would verify the typed password against the adversary's salt and hash. Furthermore, the head of the loaded list (`entries`) would be owned by a different user.

2. **Admin Account Deletion:**
   - The password manager is designed to prevent deleting the special admin user (`PWM_ADMIN_USER`). In `pwm_delete()`, a check is made to disallow the deletion:
     ```c
     if (strcmp(user, PWM_ADMIN_USER) == 0) {
       pwm_error("User '%s' cannot be deleted!", user);
       r = PWM_OPERATION_NOT_ALLOWED;
     }
     ```
   - However, the actual traversal loop for node deletion traverses the linked list starting at `entries` and checks `node->next->user`:
     ```c
     while (node -> next != NULL) {
       if (strcmp(node -> next -> user, user) == 0) {
         break;
       }
       node = node -> next;
     }
     ```
   - Because this loop checks `node->next`, **the first node (`entries`) can never be matched or deleted**.
   - If `"admin"` is in its correct first position, it is protected from deletion because it resides at the head. But if an adversary alters the position of `"admin"` (e.g., placing a dummy user at the head and moving `"admin"` to the second line), `node->next` can now match `"admin"`, allowing `pwm_delete()` to successfully remove the admin user from the memory and database.

### Solution / Code Fix
To address these vulnerabilities, we modified the `pwm_open` function in `core.c`:

1. **Enforcing Admin as the First Entry:**
   When parsing the first line (`count == 1`), we explicitly assert that the user ID is exactly `"admin"`. If it is not, we immediately abort, free the allocated node (to prevent memory leaks), and return `PWM_FILE_CORRUPT`.
   ```c
   if (count == 1) {
     if (strcmp(node->user, PWM_ADMIN_USER) != 0) {
       r = PWM_FILE_CORRUPT;
       pwm_error("Corrupt file '%s': first user must be '%s'!", (*pwm)->file, PWM_ADMIN_USER);
       free(node);
       break;
     }
     ...
   }
   ```

2. **Rejecting Empty Files:**
   If the database file is completely empty, `count` remains `0`. We added a check at the end of the load loop to verify `count > 0`, otherwise returning `PWM_FILE_CORRUPT` and closing the file.
   ```c
   if (r == PWM_OK && count == 0) {
     r = PWM_FILE_CORRUPT;
     pwm_error("Corrupt file '%s': empty password database!", (*pwm)->file);
   }
   ```

## 2. Line Duplication & User Hashing Flaw

### Vulnerability Explanation
An adversary with limited technical knowledge could duplicate a line in the password database (e.g. copying a user entry whose password is known, such as `steve`), paste it as a new line, and change only the username to a target user (e.g., `admin`). 
* This occurs because the MD5 checksum calculation previously only took into account the salt and the password:
  $$\text{Hash} = \text{MD5}(\text{salt} \mathbin{\Vert} \text{password})$$
* Because the username was entirely omitted from the hash, both entries (`admin` and `steve`) would possess identical hashes and salts. Thus, the admin's password would be successfully compromised and cloned into the system.

### Solution / Code Fix
We resolved this by binding the username cryptographically to the generated hash:

1. **Generalized Hashing Function (`utils.c`):**
   We generalized `pwm_hash_password` to accept `char* user` and prepended the username bytes to the hash buffer before hashing the salt and the password:
   ```c
   pwm_res_t pwm_hash_password(char* user, salt_t salt, char* password, hash_t checksum) {
     MD5_CTX ctx;
     MD5Init(&ctx);
     MD5Update(&ctx, (unsigned char*) user, strlen(user)); // Cryptographic binding
     MD5Update(&ctx, salt, sizeof(salt_t));
     MD5Update(&ctx, (unsigned char*) password, strlen(password));
     MD5Final(checksum, &ctx);
     return PWM_OK;
   }
   ```
   Now:
   $$\text{Hash} = \text{MD5}(\text{username} \mathbin{\Vert} \text{salt} \mathbin{\Vert} \text{password})$$
   Changing only the username on a duplicated line will now result in password validation failures, as `pwm` will hash `"admin" + salt + password`, yielding a completely different hash value.

2. **Program Logic Adaptations (`core.c` and `pwm.h`):**
   - Updated the signature of `pwm_hash_password` in `pwm.h`.
   - Updated the 4 caller locations in `core.c` to supply the respective user ID:
     - `create_node_p` $\rightarrow$ `pwm_hash_password(user, salt, password, hash)`
     - `pwm_open` $\rightarrow$ `pwm_hash_password(node->user, node->salt, password, vhash)`
     - `pwm_update` $\rightarrow$ `pwm_hash_password(node->user, salt, password, hash)`
     - `pwm_match` $\rightarrow$ `pwm_hash_password(node->user, node->salt, password, hash)`

3. **Helper Python Script Adaptation (`python_scripts/hashing_password.py`):**
   Updated the helper python tool to accept the username as its first argument and calculate the unified hash accordingly.

## 3. General Tampering Prevention: Fully Implemented AES-256-GCM AEAD

Even after binding the username to the hash and enforcing structural positions, a standard plain text database is still vulnerable. An attacker with local write privileges can arbitrarily modify record fields or perform high-speed offline dictionary attacks (e.g. using Hashcat) against the hashes.

To solve this, we went a step further than the assignment requirements and **fully implemented a production-grade Authenticated Encryption (AEAD) database protection scheme** on the dedicated branch `feat/task-4-aead`.

---

### AEAD Implementation Architecture

Our solution uses **AES-256-GCM** (via OpenSSL 3.0's EVP cipher API) and **PBKDF2-HMAC-SHA256** for secure key derivation. 

#### 1. Binary Cryptographic File Layout
When saved using the new scheme, the database is stored in a structured binary format:
```text
+----------------------+----------------------+----------------------+---------------------------+
| Magic (8B: PWMAEAD\0)| KDF Salt (16B)       | Nonce/IV (12B)       | Auth Tag (16B)            |
+----------------------+----------------------+----------------------+---------------------------+
| Ciphertext (Variable length)                                                                   |
+------------------------------------------------------------------------------------------------+
```

#### 2. Key Derivation & Ephemeral Key Management
- The 256-bit symmetric encryption key is **never saved to the disk**. It is ephemeral, resident only in RAM during execution, and derived dynamically on the fly.
- We utilize OpenSSL's `PKCS5_PBKDF2_HMAC()` with a SHA-256 digest, 10,000 iterations, the administrator's typed password, and a cryptographically secure random 16-byte KDF salt read from or written to the file header.
- Upon successful encryption or decryption, the derived key is immediately sanitized in memory (`memset` to `0`) to prevent cold-boot or memory exposure attacks.

#### 3. Tamper-Proof Encryption (`pwm_save`):
1. **Serialization:** We serialize all user records into a contiguous in-memory buffer using POSIX `open_memstream()`.
2. **Key & IV Generation:** Generate a fresh, random 16-byte KDF Salt and 12-byte Nonce/IV via OpenSSL's `RAND_bytes()`.
3. **AES-256-GCM Encryption:** Derive the key and encrypt the serialized buffer, generating the encrypted ciphertext and a 16-byte GCM authentication tag.
4. **Writing:** Write the `PWMAEAD` magic header, the cryptographic parameters (salt, IV, tag), and the ciphertext to the database file.

#### 4. Seamless Backward Compatibility & Decryption (`pwm_open`):
To ensure that standard plain text password databases (or files generated by grading test suites) are not broken, the `pwm_open` function is fully backward-compatible:
1. **Sniffing:** Reads the first 8 bytes of the database.
2. **Fallback to Plain Text:** If the file does not start with the `PWMAEAD` magic header, `pwm_open` seeks back to the beginning (`fseek(f, 0, SEEK_SET)`) and parses it directly as standard plain text.
3. **AEAD Decryption:** If `PWMAEAD` is detected, it reads the salt, IV, and tag, derives the key, and decrypts the ciphertext.
4. **POSIX Memory Stream Parsing:** The decrypted plain text string is opened as a standard file pointer (`FILE*`) in-memory using POSIX `fmemopen()`. This pointer is fed directly into the existing parser loop.
5. **Authenticity Guarantee:** Any attempt to tamper with, delete, or modify any bytes in the encrypted file invalidates GCM's authentication tag. `EVP_DecryptFinal_ex` fails immediately, rejecting the file as corrupted or notifying of password mismatch.
